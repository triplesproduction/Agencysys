CRITICAL
Server-Side Middleware Does Zero Auth Enforcement
frontend/src/middleware.ts
The Next.js middleware matches all protected routes (/dashboard, /employees, /logs, etc.) but immediately returns NextResponse.next() with no auth check whatsoever. Any unauthenticated request to these routes gets a 200 response. The only guard is client-side JavaScript (AuthGuard), which can be bypassed by disabling JS, using curl, or scraping tools.
// Current middleware — effectively does nothing return NextResponse.next(); // No auth check! // Fix — verify Supabase session server-side import { createServerClient } from '@supabase/ssr'; const supabase = createServerClient(url, key, { cookies: { ... } }); const { data: { session } } = await supabase.auth.getSession(); if (!session) return NextResponse.redirect(new URL('/login', request.url));
Fix
Use @supabase/ssr in middleware to verify the session cookie server-side before allowing access to any protected route. This is the standard Supabase + Next.js pattern.
CRITICAL
Role Escalation Via Email Address Pattern-Match
frontend/src/hooks/useAuth.tsx · lines ~60–70
In the fetchProfile fallback (when no DB record is found), the role is assigned based on whether the email contains the string "admin". Any user who signs up with an email like myadmintools@company.com or administrator@gmail.com would silently receive ADMIN privileges — including access to employee management, payroll, and system logs.
// useAuth.tsx — dangerous fallback roleId: email?.toLowerCase().includes('admin') ? 'ADMIN' : 'EMPLOYEE', // Fix — always default to the lowest privilege roleId: 'EMPLOYEE', // Role MUST come from the DB record, never inferred from email
Fix
The fallback path should always assign 'EMPLOYEE' as the role. If the DB record is missing, the user should be blocked (shown the "Profile Sync Error" screen) rather than granted any access with an inferred role. This fallback exists in three places in useAuth.tsx — all three need to be fixed.

Supabase Lock Bypass Breaks Multi-Tab Session Security
frontend/src/lib/supabase.ts
The Supabase client is configured with a custom lock function that immediately executes without acquiring any lock. The comment says this is a "workaround for AbortError" — but disabling the lock means multiple tabs can simultaneously write conflicting session state, causing unpredictable auth behavior, stale tokens being used, and potential session corruption.
// This silently disables all session locking lock: async (name, acquireTimeout, fn) => { return await fn(); } // Fix: Remove the override; investigate the AbortError root cause instead // Remove the lock property entirely and let Supabase handle it
Fix
Remove the custom lock override. The AbortError typically happens from duplicate Supabase client instantiations — ensure supabase.ts is a true singleton (it is, but verify no other createClient calls exist elsewhere).

Improvements & Optimizations
10 items
01
Add Row Level Security (RLS) Policies to Supabase
Currently, authorization is enforced entirely in frontend JavaScript, which means any user with your anon key and a SQL client can read/write all tables. Enable RLS on employees, leaves, tasks, eod_reports, messages, and kpi_profiles. Define policies like USING (auth.uid() = employee_id) so security is enforced at the DB layer regardless of how requests arrive.
Supabase Dashboard → Table Editor → RLS Policies
02
Replace axios and socket.io-client with Tree-Shaken Alternatives
Your package.json lists axios and socket.io-client as dependencies, but all networking is done through Supabase. These packages add ~50KB and ~200KB to your bundle respectively. Run npm ls axios socket.io-client to verify they are unused, then remove them. The Supabase client handles HTTP and realtime (WebSocket) natively.
frontend/package.json
03
Centralize Role Normalization — It's Duplicated ~8 Times
The pattern role.toUpperCase().includes('ADMIN') ? 'ADMIN' : ... is duplicated in useAuth.tsx (×3), dashboard/page.tsx, employees/page.tsx, api.ts, permissions.ts, and the chat function. You have getResolvedRole() in permissions.ts — use it everywhere and delete the duplicates.
frontend/src/lib/permissions.ts → getResolvedRole()
04
Extract Repeated Loading Spinner JSX into a Component
The spinning loader UI (the dark fullscreen with a rotating purple ring) is copy-pasted verbatim in AuthGuard.tsx at least 3 times with slightly different labels ("INITIALIZING", "REDIRECTING", "LOADING PROFILE"). Extract this into a <FullscreenLoader label="..." /> component and replace all instances.
frontend/src/components/auth/AuthGuard.tsx
05
Use @tanstack/react-query for Data Fetching (Already Installed)
@tanstack/react-query is in your package.json but is not used anywhere. Currently, every page re-fetches all data on every mount with no caching. Using React Query would give you automatic caching, background refetching, loading/error states, and deduplication of identical requests — all of which are currently implemented manually and inconsistently.
frontend/package.json → @tanstack/react-query@5.x
06
Dashboard Data Fetching: Use Promise.allSettled Already — but Log Rejections
The dashboard correctly uses Promise.allSettled to prevent one failing fetch from blocking the whole page. But rejected promises are silently ignored — the user sees stale 0 values with no indication something failed. Add a results.filter(r => r.status === 'rejected') check and log errors or show a subtle "Some data failed to load" banner.
frontend/src/app/dashboard/page.tsx
07
Whitelist-Based DB Payload Construction is Fragile — Use Zod or a DTO Class
In createTask and updateTask, you manually iterate over an object and filter keys against a string array whitelist. This pattern is easy to break (typo in a key name, new field not added to whitelist). Use a Zod schema or class-transformer to define the DB shape once and validate/strip fields automatically.
frontend/src/lib/api.ts · createTask(), updateTask(), submitEOD()
08
NotificationProvider Creates a Duplicate Supabase Channel ("system_events")
Both NotificationProvider.tsx and ws.ts (SupabaseRealtimeClient) subscribe to a channel named system_events. Supabase will create two separate subscriptions, and broadcast events may be delivered to both, causing duplicate notifications. Consolidate into a single channel manager.
frontend/src/lib/ws.ts · components/notifications/NotificationProvider.tsx
09
normalizeEmployee() Has a Typo in Column Name
In normalizeEmployee(), there's emp.design_ation (with an underscore in the middle of "designation"). This is almost certainly a typo and will silently fail to map the designation field correctly for some records.
frontend/src/lib/api.ts · normalizeEmployee()
10
File Uploads Use Date.now() + Math.random() — Use UUID Instead
Profile photos and documents are uploaded with filenames like profiles/1714500000000_abc123.jpg. While unlikely to collide, the better pattern for production storage is crypto.randomUUID() which gives proper v4 UUIDs. Also, files from deleted employees will remain in storage indefinitely — implement a cleanup mechanism.
frontend/src/lib/api.ts · uploadPhoto(), uploadFile(), uploadChatMedia()
